The MVT design pattern is used primarily in Django (a Python web framework) and is quite similar to MVC, but with some key differences in terminology and structure: ^mNIOys9U

Model: Similar to MVC, the Model is responsible for the data and the business logic. It manages the data and defines how the data is saved or retrieved. ^XeL7ra3I

View: In MVT, the View is different from MVC. The View is responsible for receiving user input, processing it, and returning an appropriate response (like rendering a template). It is analogous to the Controller in MVC. ^Sy6UCT41

Template: Instead of the View being responsible for displaying the data directly (like in MVC), the Template in MVT is responsible for rendering the HTML or UI elements. It’s a file that defines the structure and layout of the output. ^PWcq5QpC

The Template handles the rendering separately from business logic, 
making it easier to maintain and update the UI. ^2bmN5ru8

Views in Django are more abstracted than in MVC, making it easier to control what happens when a request is made (including middleware handling). ^BQewjEtb

Yes, using persisted connection, called CONN_MAX_AGE=None
<< limitation: one connection per one thread model.


When Django requests query from a database, the thread is left
to be IDLE. This is beucase many databases do not accept pipelining.
Essentially, the number of queries to database is limited by thread queries.

Another issue is the scheduler, lets say we have 3 worker threads, all are busy
with the database. Another client would like to connect, for a purpose that does
not require the database. What happens then? It limits performance. 
Unless we use asynchronousisation, where a thread will happily deallocated by the
kernel when required.
